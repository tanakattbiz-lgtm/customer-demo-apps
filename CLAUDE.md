# CLAUDE.md — customer-demo-apps 開発憲法

このリポジトリは、**受託開発の提案用モック(デモアプリ)を GitHub Pages で顧客に共有する**ためのものです。
ここに書かれたルールは絶対です。モックの品質 = 受注率に直結するため、**「静的サイトの制約の中で、最高にモダンで美しいデモ」**を作ることがあなたの使命です。

---

## 1. 絶対ルール(違反禁止)

1. **静的ファイルのみ。** GitHub Pages で配信されるため、サーバーサイド処理・DB・API キーが必要な外部 API は使えない。すべてブラウザだけで完結させる
2. **顧客ごとにディレクトリを分ける。** ルート直下に `顧客名/` を作り、その中だけで完結させる。他顧客のディレクトリを参照・リンクしない
3. **エントリポイントは必ず `index.html`。** ディレクトリ URL(末尾スラッシュ)だけで開けること
4. **パスは相対パスのみ。** GitHub Pages は `https://<owner>.github.io/customer-demo-apps/` のサブパス配信なので、`/assets/...` のような絶対パスは 404 になる。`./assets/...` と書く
5. **機密情報禁止。** API キー、実在の個人情報、顧客から預かった実データ、内部資料は絶対にコミットしない。データはすべて自作のダミー
6. **デモを追加したらルート `index.html` の `DEMOS` 配列に登録する。** これを忘れると一覧に出ない
7. **ルート `index.html` のパスワードゲート(`PASSWORD_HASH` と照合ロジック)を壊さない・弱めない。** パスワード平文をコード・コメント・コミットメッセージに書かない
8. **ビルドステップを持ち込まない。** npm / bundler / フレームワークの SPA ビルドは原則禁止。コミットしたファイルがそのまま動く状態を保つ(理由: 誰でも・いつでも 1 ファイル直せば直る運用にするため)

## 2. ディレクトリ・命名規則

```
customer-demo-apps/
├── index.html            # デモ一覧(パスワード保護)— DEMOS 配列で管理
├── <顧客名>/             # 英小文字ケバブケース(例: acme-corp)
│   ├── index.html        # デモ本体(単一ファイル完結が理想)
│   └── assets/           # 必要な場合のみ。画像・追加 JS/CSS はここに閉じる
└── <顧客名>/<デモ名>/    # 同一顧客に複数デモがある場合
```

- 顧客名・デモ名: 英小文字 + ハイフン(`sakura-clinic`, `acme-corp/reservation`)
- 日本語のディレクトリ名・ファイル名は禁止(URL が壊れやすい)
- 画像は軽量に(WebP/SVG 推奨、1 ファイル 500KB 以内目安)。写真が必要なら SVG イラストや CSS グラデーションで代替を検討

## 3. 技術スタック(モックの標準構成)

**原則: 1 デモ = 1 つの `index.html` に HTML/CSS/JS をすべてインライン**で書く。規模が大きい場合のみ `assets/` に分割。

### 推奨(CDN 経由・ビルド不要)

| 用途 | 推奨 | 備考 |
|---|---|---|
| CSS | **手書きのモダン CSS**(カスタムプロパティ + Grid/Flex) | 第一選択。Tailwind Play CDN も可(モック用途なら許容) |
| インタラクション | **Vanilla JS (ES2022+)** | 第一選択。宣言的に書きたい場合は Alpine.js / petite-vue |
| コンポーネント志向の大型モック | React + htm(CDN, ビルドなし)または Vue 3 CDN 版 | 本当に必要な時だけ |
| チャート | Chart.js または ECharts(CDN) | ダッシュボード系デモで |
| アニメーション | CSS transition/animation 優先、複雑なら Motion One / GSAP(CDN) | |
| アイコン | インライン SVG(Lucide / Heroicons のパスをコピー) | アイコンフォント禁止 |
| フォント | Google Fonts: 日本語は `Noto Sans JP` / `Zen Kaku Gothic New`、英数は `Inter` / `Manrope` | `display=swap` を付ける |

### モダン CSS/JS を積極的に使う(2025+ 基準・主要ブラウザ最新版対象)

- CSS: ネスト、`:has()`、`color-mix()`、`oklch()`、コンテナクエリ、`clamp()` による流体タイポグラフィ、`scroll-timeline` / `animation-timeline`(プログレッシブエンハンスメントとして)、`view-transition`、`backdrop-filter`
- JS: ES Modules(`<script type="module">`)、`fetch`、`IntersectionObserver`、`dialog` 要素、Web Crypto、`localStorage`(デモ内の状態永続化に活用し「動いてる感」を出す)
- 古いブラウザ対応・IE 対応は**一切不要**

## 4. デザイン基準(ここで差がつく)

モックは「動く仕様書」ではなく**「顧客の心を動かすプロダクトの未来図」**。テンプレ感のある Bootstrap 風デザインは禁止。毎回、その顧客の業種・ブランドに合わせたデザインを設計する。

### 4.1 必須要件

- **レスポンシブ必須**(顧客はスマホで開くことが多い。モバイルで崩れたら即失注)
- `<html lang="ja">`、適切な `<title>`、`viewport` メタ、OGP は不要(むしろ検索避けに `<meta name="robots" content="noindex">` を必ず入れる)
- 表示は日本語(顧客向けのため)。UI 文言・ダミーデータも自然な日本語で

### 4.2 デザインシステムを最初に定義する

各デモの `<style>` 冒頭で必ずデザイントークンを定義してから書き始める:

```css
:root {
  /* 顧客の業種・ブランドから選定する */
  --color-primary: oklch(55% 0.2 260);
  --color-surface: ...;
  --color-text: ...;
  --radius-md: 12px;
  --shadow-md: 0 4px 24px oklch(0% 0 0 / 0.08);
  --font-body: "Noto Sans JP", sans-serif;
  --space-unit: 4px; /* 余白は 4px グリッド */
}
```

- **配色**: プライマリ 1 色 + ニュートラル階調 + セマンティック色(成功/警告/エラー)。彩度の高い色の多用禁止。業種に合わせる(医療=清潔な青緑、飲食=温かい暖色、SaaS=知的な青紫 など)
- **タイポグラフィ**: 見出しと本文でウェイト差を明確に。`clamp()` で流体サイズ。行間 1.6〜1.8(日本語)
- **余白**: 詰めない。セクション間は大胆に(80〜120px)。カード内パディングは 24〜32px
- **角丸・影**: トーンを統一(全体でシャープ or 全体でソフト)。影は薄く多層に

### 4.3 モダンな表現の引き出し(適材適所で使う)

- ガラスモーフィズム(`backdrop-filter: blur()`)、ベントーグリッド、大胆なヒーローセクション、グラデーションメッシュ背景、スクロール連動アニメーション、マイクロインタラクション(hover/focus/押下の反応)、スケルトンローディング、数値のカウントアップ演出
- ダークモード対応(`prefers-color-scheme` + トークン切り替え)は SaaS・管理画面系デモで加点要素
- アニメーションは**上品に短く**(150〜400ms、`ease-out` 系)。`prefers-reduced-motion` を尊重

### 4.4 「動くモック」にする

静的な絵ではなく、触れるプロトタイプにする:

- ボタン・タブ・モーダル・フォームは実際に動かす(送信は `localStorage` 保存 + 成功トーストで擬似再現)
- 一覧画面には現実的な件数(8〜20 件)の**リアルなダミーデータ**を用意(「テスト1」「あああ」禁止。実在しそうな名前・日付・金額を作る)
- 検索・絞り込み・並び替えなど、提案の肝になる機能は JS で本当に動かす
- 画面遷移が複数ある場合は 1 ファイル内のビュー切り替え(hash ルーティング)か、同ディレクトリ内の複数 HTML で

## 5. デモ一覧への登録(必須手順)

新しいデモを作ったら、ルート `index.html` 内の `DEMOS` 配列に追加する:

```js
const DEMOS = [
  {
    customer: "acme-corp",        // ディレクトリ名
    name: "ACME 株式会社",         // 顧客表示名
    title: "予約管理システム",      // デモのタイトル
    path: "./acme-corp/",          // 相対パス(末尾スラッシュ)
    date: "2026-07-08",            // 作成日
    tags: ["予約", "管理画面"],     // 任意
    status: "active",              // active | archived
  },
];
```

## 6. 品質チェックリスト(コミット前に必ず確認)

- [ ] `python3 -m http.server` 等でローカル配信し、デモが動くことを実際に確認した
- [ ] スマホ幅(375px)・タブレット幅(768px)・PC 幅(1280px)で崩れない
- [ ] すべてのリンク・アセットが相対パス
- [ ] `<meta name="robots" content="noindex">` を入れた
- [ ] コンソールエラーが出ていない
- [ ] ダミーデータに実在の個人情報・機密が混ざっていない
- [ ] ルート `index.html` の `DEMOS` に登録した
- [ ] 他顧客のディレクトリに一切触れていない

## 7. Git 運用

- コミットメッセージは日本語可。`add: acme-corp 予約システムのモック追加` のように何をしたか明確に
- `main` へのプッシュ = 即公開と同義。中途半端な状態をプッシュしない
- パスワードや機密がコミット履歴に入った場合は、履歴書き換えが必要になる。**そもそも入れない**

## 8. パスワードゲートの仕様(ルート index.html)

- 入力パスワードを Web Crypto (`crypto.subtle.digest("SHA-256", ...)`) でハッシュ化し、定数 `PASSWORD_HASH` と照合
- 認証成功は `sessionStorage` に保持(ブラウザを閉じるとリセット)
- **顧客の各デモページにはゲートを付けない**(一覧ページのみ)
- パスワード変更手順: `printf '新パスワード' | shasum -a 256` で生成したハッシュに `PASSWORD_HASH` を差し替える
- これはクライアントサイドの簡易ゲートであり、真のアクセス制御ではない。だからこそルール 5(機密情報禁止)が絶対
